0.1 核心设计哲学
-----------------
本项目遵循以下核心设计原则：

**0.1.1 配置优先于代码**
- 原因：参数需要频繁调整，编译太慢
- 实施：所有可调参数都通过YAML配置
- 例外：物理常量（装甲板尺寸）、系统设计常量（重试次数）保留在代码中
- 禁止：在代码中硬编码可调参数（如速度、阈值等）

**0.1.2 后备机制优先于简化**
- 原因：配置文件可能丢失或损坏，系统必须能运行
- 实施：YAML有默认值，代码有后备常量
- 优先级：YAML > 代码默认值 > 代码后备常量
- 禁止：删除代码中的后备常量

**0.1.3 模块化优先于单体**
- 原因：功能独立，易于测试和维护
- 实施：每个类职责单一，接口清晰
- 禁止：在一个类中混合导航、射击、检测等多种职责

**0.1.4 性能优先于抽象**
- 原因：机器人控制需要实时响应
- 实施：关键路径使用原生类型，避免不必要的封装
- 禁止：在热路径中使用虚函数或动态分配

0.2 关键架构决策
-----------------

**0.2.1 为什么ConfigManager而不是修改代码？**
- 问题：调整速度参数需要重新编译（20秒）
- 解决：YAML配置，修改后立即生效（需要重启程序）
- 设计：
  * GoNode加载ConfigManager
  * ConfigManager传递给所有核心类
  * 每个类在构造函数中读取配置到成员变量
  * 运行时使用成员变量，避免重复解析YAML
- 陷阱：不要在热路径中调用ConfigManager，应在构造函数中缓存

**0.2.2 为什么有3个定时器而不是1个？**
- control_timer（5ms）：主控制循环，导航和状态切换
- shoot_timer（30ms）：自瞄计算，不需要1ms更新
- swing_timer（20ms）：摆动角度更新和射击命令发布
- 原因：
  * 不同任务需要不同频率，避免浪费CPU
  * 自瞄计算较慢，30ms足够
  * 射击命令需要稳定频率，20ms平滑
- 禁止：合并定时器或改变定时器频率

**0.2.3 为什么摆动用20ms而不是1ms更新？**
- 原因：1ms太快会导致射击命令抖动
- 20ms的好处：
  * 减少射击命令频率（从1000Hz降到50Hz）
  * 平滑摆动过程，避免机械震动
  * 降低通信负载
- 禁止：修改swing_timer的频率

**0.2.4 为什么D2C2每帧只保留最高置信度目标？**
- 原因：多目标会导致云台在目标间跳变
- 实施：std::sort后resize(1)
- 陷阱：不要保留所有目标，必须单目标

**0.2.5 为什么滤波器alpha=0.1而不是0.3？**
- 原因：0.3太快，会导致角度抖动
- 0.1的好处：
  * 慢响应，减少抖动
  * 稳定跟踪目标
- 陷阱：不要随意修改alpha，可能导致失稳

**0.2.6 为什么shoot_timer是30ms而不是1ms？**
- 原因：1ms太频繁，自瞄计算不需要这么快
- 30ms的好处：
  * 减少CPU占用（从1000Hz降到33Hz）
  * 相机图像30Hz，30ms足够
- 禁止：修改shoot_timer的频率

**0.2.7 为什么使用策略模式实现路径规划？**
- 原因：Player1和Player2路径不同，但逻辑相同
- 实施：IPathPlanningStrategy接口 + 工厂方法
- 好处：
  * 添加新玩家只需新增策略类
  * 代码复用，避免重复
- 陷阱：不要在策略中硬编码路径，应从ConfigManager读取

**0.2.8 为什么使用工厂模式创建策略？**
- 原因：根据player_id动态选择策略
- 实施：PathStrategyFactory::createStrategy(player_id)
- 好处：解耦创建逻辑，易于测试

**0.2.9 为什么ConfigManager使用指针传递？**
- 原因：避免拷贝大型配置对象
- 实施：ConfigManager* config_manager_
- 陷阱：确保ConfigManager生命周期长于使用它的类

**0.2.10 为什么删除了44行硬编码常量？**
- 原因：这些常量都是可调参数，不应该硬编码
- 保留的常量：
  * 物理常量（MAP_UNIT_TO_METER、ARMOR_SIZE）
  * 系统设计常量（MAX_RETRY_COUNT、PATH_COUNT）
  * 算法常量（TOLERANCE、MAX_REASONABLE）
- 删除的常量：
  * 速度参数（MAX_SPEED_*）
  * 摆动参数（SWING_*）
  * 相机参数（CAMERA_*）
  * 弹道参数（BULLET_SPEED、GRAVITY）
  * 阈值参数（ARRIVAL_THRESHOLD_*）

0.4 常见陷阱与错误模式
-----------------

**0.4.1 ConfigManager使用陷阱**
❌ 错误：在热路径中调用config_manager_->getConfig()
✅ 正确：在构造函数中缓存到成员变量
原因：每次调用都会解析YAML，性能极差

❌ 错误：ConfigManager生命周期短于使用它的类
✅ 正确：ConfigManager是GoNode的成员变量，生命周期最长
原因：悬空指针导致崩溃

**0.4.2 定时器陷阱**
❌ 错误：在回调中使用阻塞操作（sleep、文件IO）
✅ 正确：使用异步操作或快速返回
原因：阻塞会阻塞整个定时器线程

❌ 错误：修改定时器频率（如从5ms改为10ms）
✅ 正确：保持现有频率，如需优化使用异步任务
原因：定时器频率经过性能调优，改变会导致系统失稳

**0.4.3 滤波器陷阱**
❌ 错误：alpha值过大（如0.5）
✅ 正确：alpha=0.1，需要更慢则改为0.05
原因：alpha过大导致抖动，alpha过小导致响应慢

❌ 错误：在滤波器未初始化时使用
✅ 正确：首次使用前先update一次
原因：未初始化的滤波器返回无效值

**0.4.4 状态机陷阱**
❌ 错误：状态转换时忘记重置相关变量
✅ 正确：状态转换时重置所有相关状态
原因：状态残留导致异常行为

❌ 错误：直接修改state_而不是通过状态转换函数
✅ 正确：使用条件判断触发状态转换
原因：直接修改会绕过状态转换逻辑

**0.4.5 内存管理陷阱**
❌ 错误：使用裸指针管理资源
✅ 正确：使用std::unique_ptr或std::shared_ptr
原因：裸指针容易内存泄漏

❌ 错误：在ROS2回调中使用阻塞操作
✅ 正确：使用异步任务或快速返回
原因：阻塞会阻塞整个ROS2线程

**0.4.6 性能陷阱**
❌ 错误：在热路径中使用虚函数
✅ 正确：使用模板或内联函数
原因：虚函数有额外开销

❌ 错误：在热路径中使用动态内存分配
✅ 正确：使用栈分配或对象池
原因：new/delete开销大，可能导致碎片

**0.4.7 多线程陷阱**
❌ 错误：在多个定时器回调中访问共享变量而不加锁
✅ 正确：使用原子变量或互斥锁
原因：数据竞争导致不可预测行为

❌ 错误：在回调中修改ConfigManager
✅ 正确：ConfigManager只读，不要修改
原因：多线程写入导致竞争条件

**0.4.8 编码陷阱**
❌ 错误：使用魔法数字（如42、3.14159）
✅ 正确：使用命名常量或从ConfigManager读取
原因：魔法数字难以维护和理解

❌ 错误：不检查返回值（如文件打开、解析结果）
✅ 正确：检查所有可能的错误情况
原因：忽略错误会导致难以调试的问题

0.8 紧急情况处理
-----------------

**0.8.1 程序无法启动**
1. 检查模型文件：ls /home/robot/competition/models/
2. 检查配置文件：ls /home/robot/competition/src/go/config/
3. 检查编译产物：ls /home/robot/competition/install/go/lib/go/
4. 检查ROS2环境：echo $ROS_DISTRO
5. 恢复备份：cp go_d2c2.cpp.bak go_d2c2.cpp

**0.8.2 编译失败**
1. 检查语法错误
2. 检查依赖是否安装
3. 检查CMakeLists.txt配置
4. 清理构建产物：rm -rf build install log
5. 重新编译：colcon build

**0.8.3 运行时崩溃**
1. 查看崩溃日志：dmesg | tail -50
2. 使用GDB调试：gdb ./install/go/lib/go/S_d2c2
3. 检查内存泄漏：valgrind --leak-check=full ./install/go/lib/go/S_d2c2 1
4. 恢复备份：cp go_d2c2.cpp.bak go_d2c2.cpp
5. 重新编译测试

**0.8.4 配置文件加载失败**
1. 检查YAML语法：python3 -c "import yaml; yaml.safe_load(open('game_config.yaml'))"
2. 检查文件路径
3. 检查文件权限
4. 检查yaml-cpp是否安装：dpkg -l | grep yaml-cpp

**0.8.5 性能突然下降**
1. 检查CPU占用：top -p $(pgrep -f "ros2 run go")
2. 检查推理延迟：查看性能监控报告
3. 检查是否有内存泄漏：valgrind
4. 检查是否有死锁：pstack $(pgrep -f "ros2 run go")

2.2.5 CenterNetDetector - 深度学习检测类
-----------------
功能：基于 TensorRT GPU 加速的深度学习目标检测

检测模型：
• 模型名称：CenterNet（目标检测）
• 推理引擎：TensorRT FP16
• 输入尺寸：224×416 RGB
• 输出尺寸：56×104 热力图
• 类别数：2类（红方/蓝方）
• 模型文件：models/centernet_r18d2c2_fp16.engine
• 训练代码：D2C2/train/train_2class.py
• 推理延迟：约0.38ms（GPU）

D2C2模型特点：
• 轻量级：使用ResNet18骨干网络
• 高精度：自适应高斯半径，数据增强
• 稳定性：验证集监控，Early Stopping
• 实时性：FP16量化，低延迟推理
• 置信度约束：推理时自动clip到[0, 1]范围

检测流程：
1. 图像预处理（调整尺寸、RGB转换、归一化）
2. GPU 推理（TensorRT 加速）
3. 热力图解码（Top-K 排序）
4. 边界框解码（回归框+宽高）
5. 目标过滤（类别过滤、置信度阈值）
6. 置信度排序（只保留最高置信度目标）
7. PnP 距离解算
8. 运动预测和弹道补偿

主要方法：
• initTensorRT() - 初始化 TensorRT 引擎
• updateFrame() - 更新图像帧
• detectArmors() - 检测装甲板
• imageToGimbalAngles() - 图像坐标到云台角度转换
• calculateBallisticCompensation() - 弹道补偿计算（带滤波）
• solvePnPDistance() - PNP距离解算（带距离滤波）
• predictTargetPosition() - 目标位置预测（5帧历史）
• updateTargetHistory() - 更新目标位置历史
• resetFilters() - 重置所有滤波器

相机参数：
• 焦距：Fx=Fy=554.26（统一焦距）
• 光心：Cx=320.0, Cy=240.0
• 图像分辨率：640×480
• 装甲板尺寸：200mm×125mm

识别类别配置（D2C2两分类模型）：
• 类别 0：红方装甲板
• 类别 1：蓝方装甲板
• Player1（蓝队）: 识别类别 0（红方）- 识别敌方红色
• Player2（红队）: 识别类别 1（蓝方）- 识别敌方蓝色
• 每帧只保留最高置信度的目标（避免多目标干扰）

滤波系统：
- MovingAverageFilter: 距离滤波（窗口大小5）
- ExponentialSmoothingFilter: yaw/pitch 滤波（alpha=0.1）


具体策略：
• Player1PathStrategy - Player1的路径点顺序
• Player2PathStrategy - Player2的路径点顺序

工厂方法：
```cpp
class PathStrategyFactory {
public:
    static std::unique_ptr<IPathPlanningStrategy> createStrategy(int player_id);
};
```

使用方式：
```cpp
auto strategy = PathStrategyFactory::createStrategy(player_id_);
auto path_indices_ = strategy->generatePathIndices();
```

优势：
• 易于扩展：添加新玩家只需新增策略类
• 符合开闭原则：对扩展开放，对修改关闭
• 代码复用：策略之间共享 Waypoint 结构体
• 逻辑清晰：路径规划逻辑独立封装

2.2.8 PerformanceMonitor - 性能监控类（新增）
-----------------
功能：实时监控系统性能指标

监控指标：
• 推理延迟（inference）
• 平均值、最小值、最大值、标准差
• 采样数量

主要方法：
• startTimer() - 开始计时
• endTimer() - 结束计时并记录
• recordValue() - 记录自定义值
• getMetrics() - 获取指标统计信息
• reportAll() - 输出所有指标统计信息
• reset() - 清空所有指标
• getMetricNames() - 获取所有指标名称

报告周期：每10秒输出一次

输出示例：
```
========== 性能监控报告 ==========
inference: avg=1.79ms, min=1.30ms, max=3.06ms, std=0.36ms (采样: 307)
=================================
```

优势：
• 实时监控：随时了解系统性能
• 统计分析：提供详细的统计信息
• 灵活记录：支持自定义指标
• 易于扩展：可添加新的监控指标

2.2.9 GoNode - 主节点类
-----------------
功能：系统的核心控制节点，整合所有模块实现完整功能

继承：rclcpp::Node（ROS2节点基类）

ROS2话题订阅：

ROS2话题发布：

定时器：

2.3 状态机设计

2.3.1 MOVING 状态
-----------------
功能：在点位之间移动

行为：
• 计算到目标点位的方向和距离
• 根据距离自适应调整速度
• 检测到达条件
• 处理超时和重试

速度控制策略：
• 远距离（>10地图单位）：50地图单位/s（10m/s）
• 中距离（1-10地图单位）：15地图单位/s（3m/s）
• 近距离（<1地图单位）：4地图单位/s（0.8m/s）

到达条件：
• P1-P3：距离<2.0地图单位（快速到达）
• P4-P11：距离<0.8地图单位（正常到达）
• 精确调整：距离<0.2地图单位（精确到达）

2.3.2 SHOOTING 状态
-----------------
功能：在射击点位执行射击任务

行为：
• 启用射击控制器
• 根据射击模式更新云台角度
• 监控血量变化
• 检测被攻击
• 判断目标是否被摧毁

射击模式切换：
自瞄 → 硬编码 → 摆动 →（血量变化达标）→ 自瞄

2.3.3 状态转换条件
-----------------
MOVING → SHOOTING：
• 到达有射击任务的点位
• 距离小于到达阈值

SHOOTING → MOVING：
• 目标被摧毁（血量<=0）
• 自身被攻击（根据点位不同有不同策略）
• SHOOTING状态下被推得太远（等待5秒后判定）

4.6.1 性能监控报告
-----------------
系统会每10秒输出一次性能监控报告：

```

```

指标说明：

6.0 D2C2模型训练与导出

6.0.1 D2C2模型架构
-----------------
D2C2模型是基于CenterNet的两分类装甲板检测模型，专为红/蓝装甲板识别设计。

网络结构：
- 骨干网络：ResNet18（轻量级）
- 输入尺寸：224×416 RGB
- 输出尺寸：56×104（下采样4倍）
- 输出分支：
  • fmap（热力图）：2通道（红方/蓝方）
  • reg（位置偏移）：2通道（x, y offset）
  • wh（宽高）：2通道（width, height）

6.0.2 训练配置
-----------------
训练参数：
- 批次大小：8
- 训练轮数：100（Early Stopping自动停止）
- 学习率：1e-4
- 优化器：Adam
- 学习率调度：每40轮降低10倍
- 数据集分割：训练集8000张，验证集2000张（8:2）

损失函数：
- 热力图损失：Focal Loss（alpha=0.25, gamma=2）
- 位置偏移损失：L1 Loss（权重0.1）
- 宽高损失：L1 Loss（权重0.1）
- 总损失：fmap_loss + 0.1×reg_loss + 0.1×wh_loss

数据增强：
- 随机水平翻转（50%概率）
- 随机颜色抖动（HSV空间）

关键改进：
- 自适应高斯半径：根据目标大小动态调整
- 验证集监控：防止过度拟合
- Early Stopping：15轮无改善自动停止
- 置信度约束：推理时clip到[0, 1]

6.2 PnP距离计算算法

6.2.1 算法原理
-----------------
PNP（Perspective-n-Point）是一种根据n个3D-2D对应点对求解相机位姿的算法。

输入：
• 2D点：装甲板边界框的4个角点（图像坐标）
• 3D点：装甲板在世界坐标系中的实际尺寸
• 相机内参矩阵：K

输出：
• 旋转向量：rvec
• 平移向量：tvec
• 距离：tvec[2]（Z方向距离）

6.2.2 装甲板3D模型
-----------------
装甲板尺寸（真实世界）：
• 宽度：200mm = 1.0地图单位
• 高度：125mm = 0.625地图单位

3D角点坐标（以装甲板中心为原点）：
• 左上：(-0.5, -0.3125, 0)
• 右上：(0.5, -0.3125, 0)
• 左下：(-0.5, 0.3125, 0)
• 右下：(0.5, 0.3125, 0)

6.2.3 相机内参矩阵
-----------------
K = [fx,  0, cx]
    [ 0, fy, cy]
    [ 0,  0, 1]

其中：
• fx = fy = 554.26（统一焦距）
• cx = 320.0
• cy = 240.0

6.2.4 距离滤波
-----------------
使用移动平均滤波器平滑 PnP 距离:
- 窗口大小：5
- 滤波方式：最近5次距离的平均值
- 目的：减少 PnP 距离的抖动

6.2.5 距离计算结果
-----------------
成功：返回滤波后的距离
失败：返回默认值3.0米

6.3 弹道补偿算法

6.3.1 理论基础
-----------------
考虑重力和飞行时间的弹道方程：
• 水平运动：x = v_x × t
• 垂直运动：z = v_z × t - 0.5 × g × t²

6.3.2 补偿计算步骤
-----------------
1. 计算飞行时间：
   t_flight = distance / bullet_speed
   其中：bullet_speed = 23m/s

2. 计算重力下落：
   gravity_drop = 0.5 × g × t_flight²
   其中：g = 5地图单位/s²

3. 计算俯仰补偿角：
   pitch_compensation = asin(gravity_drop / distance) × 180 / π

4. 滤波补偿：
   只滤波偏移量，不使用 current_yaw 作为基础
   filtered_yaw_offset = yaw_filter_.update(yaw_offset)
   filtered_pitch_offset = pitch_filter_.update(pitch_offset + pitch_compensation)

5. 计算最终角度：
   final_yaw = current_yaw + filtered_yaw_offset
   final_pitch = current_pitch + filtered_pitch_offset

6.3.3 补偿限制
-----------------
- 限制 asin 参数范围在 [-0.99, 0.99]
- 避免小距离时不稳定

6.3.4 弹道参数
-----------------
- 子弹速度：23 m/s
- 重力加速度：5 地图单位/s²
- 滤波器 alpha：0.1（慢响应，减少抖动）

6.4 滤波系统

6.4.1 移动平均滤波器（MovingAverageFilter）
-----------------
用途：平滑 PnP 距离，减少抖动

算法：
- 窗口大小：5
- 滤波方式：最近5次距离的平均值
- 更新方式：新值加入队列,移除最旧的值

公式：
```
filtered_value = (v1 + v2 + v3 + v4 + v5) / 5
```

6.4.2 指数平滑滤波器（ExponentialSmoothingFilter）
-----------------
用途：平滑 yaw 和 pitch 角度,减少跳变

算法：
- 滤波公式：filtered = α × new_value + (1-α) × filtered
- 滤波系数：α = 0.1（慢响应）
- 初始化：首次更新直接使用新值，之后使用公式

公式：
```
filtered_value = 0.1 × new_value + 0.9 × old_value
```

6.5 运动预测算法

6.5.1 原理
-----------------
基于历史位置的线性外推预测。

6.5.2 算法步骤
-----------------
1. 获取最近5帧位置
2. 计算平均速度（像素/秒）
3. 预测下一帧位置
4. 返回预测中心点

6.5.3 历史记录管理
-----------------
- 最大历史帧数：5帧
- 预测时间间隔：0.01秒
- 采样间隔：约33ms（30Hz）

6.5.4 边界情况处理
-----------------
- 历史记录为空：返回图像中心 (320, 240)
- 历史记录只有1个：返回这个点
- 历史记录≥2个：正常进行运动预测

6.6 血量变化计算

6.6.1 滑动窗口
-----------------
使用5秒滑动窗口计算血量变化。

窗口维护：
• 新数据：添加到队列末尾
• 过期数据：移除超过5秒的数据
• 计算：当前血量 - 5秒前血量

6.6.2 变化判定标准
-----------------
各点位的血量变化阈值：

| 点位 | 目标 | 阈值 | 说明 |
|------|------|------|------|
| P4 | 3号/2号兵 | -50 | 血量减少>50算达标 |
| P5 | 2号/1号兵 | 0 | 任何减少都算达标 |
| P6 | 前哨站 | -60 | 血量减少>60算达标 |
| P7-P11 | 基地 | -120 | 血量减少>120算达标 |

注意：阈值是负数，表示血量减少。实际判定时，血量变化 < 阈值才算达标。

6.7 速度控制算法

6.7.1 三段式速度控制
-----------------
根据距离分为三个速度区域：

1. 远距离区域（distance > speed_switch_distance）：
   速度 = MAX_SPEED_FAR = 50地图单位/秒

2. 中距离区域（PRECISE_MODE_DISTANCE < distance <= speed_switch_distance）：
   速度 = MAX_SPEED_NEAR = 15地图单位/秒

3. 近距离区域（distance <= PRECISE_MODE_DISTANCE）：
   速度 = MAX_SPEED_PRECISE + (MAX_SPEED_NEAR - MAX_SPEED_PRECISE) × (distance / PRECISE_MODE_DISTANCE)
   范围：4 ~ 15地图单位/秒（线性过渡）

6.7.2 点位差异化控制
-----------------
P1-P3（快速点位）：
• speed_switch_distance = 10地图单位
• arrival_threshold = 2地图单位
• 精确调整：不进行

P4-P11（普通点位）：
• speed_switch_distance = 5地图单位
• arrival_threshold = 0.8地图单位
• 精确调整：距离 < 0.2地图单位时触发

6.7.3 精确调整
-----------------
触发条件：
• 到达目标点位但距离 > 0.2地图单位
• 仅在P4-P11点位执行

调整速度：
• 基础速度：PRECISE_ADJUST_SPEED = 1.5地图单位/秒
• 速度比例：distance / 0.2
• 实际速度：1.5 × min(1.0, distance / 0.2)

6.8 摆动算法

6.8.1 摆动方程
----------------_
yaw(t) = base_yaw + swing_direction × swing_speed × t

边界条件：
• yaw > +6度：设置为+6度，方向取反
• yaw < -6度：设置为-6度，方向取反

6.8.2 参数
----------------_
• 摆动范围：±6度
• 摆动速度：10度/秒
• 更新频率：1000Hz（1ms）
• 单次更新：10 × 0.001 = 0.01度

6.8.3 摆动停止条件
----------------_
• 检测到血量变化达标
• 恢复到前一秒的Yaw/Pitch角度

9.0 D2C2完整工作流程
================================================================================

9.1 从零开始训练D2C2模型
-----------------
步骤1：准备数据集
```bash
# 确保数据集在正确位置
cd /home/robot/competition/D2C2/datasets
ls RMCOCO2/  # 应包含images/和annotations/
```

步骤2：安装依赖
```bash
cd /home/robot/competition/D2C2/train
pip install -r requirements_train.txt
```

步骤3：训练模型
```bash
python3 train_2class.py
# 训练会自动：
# - 从checkpoint恢复（如果存在）
# - 每20轮保存checkpoint
# - Early Stopping自动停止
# - 训练完成后自动导出ONNX
```

步骤4：转换为TensorRT
```bash
cd ../scripts
bash convert_to_tensorrt.sh
# 输出：../inference/centernet_r18d2c2_fp16.engine
```

步骤5：复制到模型目录
```bash
cp ../inference/centernet_r18d2c2_fp16.engine ../../models/
```

步骤6：重新编译
```bash
cd /home/robot/competition
colcon build --packages-select go
source install/setup.bash
```

步骤7：运行测试
```bash
ros2 run go go_d2c2 1  # Player1
ros2 run go go_d2c2 2  # Player2
```

9.2 D2C2项目目录说明
-----------------
```
D2C2/
├── train/                    # 训练库
│   ├── train_2class.py    # 训练脚本
│   ├── requirements_train.txt  # Python依赖
│   └── models/            # 训练输出
│       ├── centernet_2class_best.pth  # 最佳模型
│       ├── centernet_2class.onnx     # ONNX模型
│       └── centernet_2class_epoch*.pth  # Checkpoints
├── inference/                # 推理库
│   └── centernet_r18d2c2_fp16.engine  # TensorRT引擎
├── datasets/                 # 数据集
│   ├── RMCOCO2/            # 训练数据集
│   ├── RMRobot/            # 机器人数据集
│   └── MSCOCO/             # COCO数据集
├── scripts/                  # 工具脚本
│   └── convert_to_tensorrt.sh  # TensorRT转换
└── README.md                # 使用说明
```

9.3 模型对比总结
-----------------
| 特性 | r18d4c6 | r18d2c6 | D2C2 (推荐) |
|------|---------|---------|-------------|
| 骨干网络 | ResNet18 | ResNet18 | ResNet18 |
| 输入尺寸 | 416×224 | 416×224 | 224×416 |
| 输出尺寸 | 104×56 | 208×112 | 56×104 |
| 类别数 | 6 | 6 | 2 |
| 推理延迟 | ~5ms | ~10ms | **0.38ms** |
| 识别精度 | 中 | 高 | 高 |
| 专用性 | 通用 | 通用 | **红/蓝装甲板** |
| 训练难度 | 高 | 高 | **低（两分类）** |
| 推荐场景 | 旧系统 | 特殊场景 | **正常比赛** |

9.4 D2C2模型优势总结
-----------------
1. **轻量高效**：ResNet18 + 56×104输出，推理仅0.38ms
2. **专用优化**：专门针对红/蓝装甲板训练，准确率高
3. **稳定可靠**：置信度约束，单目标过滤，避免异常
4. **易于训练**：两分类简单任务，训练快速（1.5小时）
5. **完整流程**：从训练到推理全流程支持
6. **验证保障**：Early Stopping防止过度拟合

9.7 性能监控使用流程
-----------------
步骤1：启动程序
```bash
ros2 run go go_d2c2 1
```

步骤2：查看性能报告
```bash
# 每10秒会自动输出性能报告
========== 性能监控报告 ==========
inference: avg=1.79ms, min=1.30ms, max=3.06ms, std=0.36ms (采样: 307)
=================================
```

步骤3：分析性能指标
• avg - 平均值，查看整体性能
• min - 最小值，查看最佳情况
• max - 最大值，查看最差情况
• std - 标准差，查看稳定性
• 采样 - 采样数量，确保数据充足

步骤4：优化性能
• 如果平均延迟过高，考虑优化模型
• 如果标准差过大，考虑增加滤波
• 如果最大值异常，检查系统负载
